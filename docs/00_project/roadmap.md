# プロダクトロードマップ

TODOアプリ(CRUD)+ 写経可能な教材ドキュメント一式を作る計画の全体像です。
各スプリントの詳細計画は `docs/0X_sprintX/` 配下のスプリントバックログで確定させます。

- **第1部 新規開発編(スプリント1〜4)**: ✅ 完了(2026-07-04)。ゼロからCRUD完成まで
- **第2部 追加開発編(スプリント5〜8)**: ✅ 完了(2026-07-04)。期限日・絞り込み・
  ページネーション・自動テスト/CI・認証・認可までを、既存を壊さず積み増した

## ゴール

- **プロダクトゴール**: Laravel Sail 上で動く、CRUD機能を持つTODOアプリ
- **教材ゴール**: 初級エンジニアが一人で写経・追体験できるドキュメント一式
  (開発トレースガイド・概念解説集・スクラム成果物)

## 前提

- 写経者の学習時間: **2〜3時間/スプリント**(→ [qa-log.md](qa-log.md) 参照)
- 技術スタック: Docker + Laravel Sail / Laravel / Plain HTML/CSS(Blade可)/ MySQL

## スプリント計画(第1部)

| スプリント | テーマ | 主な内容 | 写経目安 |
|---|---|---|---|
| Sprint 1 | 環境構築と土台 | Sailで環境構築、.env整備、Welcome画面表示、GitHub整備、README(鶏と卵問題の解決手順含む) | 1.5〜2h |
| Sprint 2 | Read(一覧・詳細) | マイグレーション、Todoモデル、シーダー、一覧表示、詳細表示、tinkerでのモデル検証 | 2.5〜3h |
| Sprint 3 | Create(新規作成) | 作成フォーム、バリデーション、CSRF、PRGパターン、フラッシュメッセージ | 2.5〜3h |
| Sprint 4 | Update / Delete と仕上げ | 編集、完了トグル、削除(確認ページ方式)、スタイル調整、教材ドキュメント最終化 | 2.5〜3h |

## 各スプリントの完了条件(サマリ)

- **Sprint 1**: クローン直後の人がREADMEの手順だけで `http://localhost:8080` にWelcome画面を表示できる
- **Sprint 2**: シーダー投入したTODOが一覧・詳細画面で閲覧できる(存在しないIDは404)
- **Sprint 3**: フォームからTODOを作成でき、バリデーションエラー時は入力値を保持して差し戻される
- **Sprint 4**: 編集・完了切り替え・削除(確認ページ経由)ができ、全教材ドキュメントが完成している

## なぜこの順番か(初級者向け解説)

1. **環境が最初**: 動く土台がないと何も検証できない。また環境構築は教材の最初のつまずきポイントなので、独立したスプリントとして丁寧に扱う
2. **ReadがCreateより先**: 「データを表示する」は「データを作る」より部品が少ない。
   シーダー(プログラムでダミーデータを入れる仕組み)を使えばフォームなしでデータを用意でき、
   MVC の流れ(ルート→コントローラ→モデル→ビュー)を最小構成で学べる
3. **CreateがUpdateより先**: Update は「Read(既存値の表示)+ Create(フォーム送信)」の合成技。
   部品を先に揃えてから組み合わせる
4. **Deleteが最後**: 一番シンプルに見えて「GETでデータを変えない原則」「確認ステップ」という
   設計判断が詰まっている。他のCRUDを学んだ後の方が理解しやすい

# 第2部: 追加開発編(スプリント5〜8)

新規開発の追体験(第1部)に続き、**動いているアプリへの追加開発**を追体験する続編。
テーマは「初級→実務の階段」— 既存コードを読み、既存データを壊さず、
既存機能をリグレッションさせずに変更する練習をする。

トレースガイド第2部の各ステップには、第1部になかった3要素
(**影響調査 / 既存データへの配慮 / リグレッション確認**)を含める。

## スプリント計画(第2部)

| スプリント | テーマ | 主な内容 | 写経目安 |
|---|---|---|---|
| Sprint 5 | Readの発展 | 期限日(due_date)追加、状態・キーワード絞り込み、ページネーション、SQLインジェクション疑似体験 | 2.5〜3h |
| Sprint 6 | 自動テストとCI | CRUD一式のFeatureテスト、Todoファクトリ、GitHub ActionsでPR時自動テスト | 2.5〜3h |
| Sprint 7 | 認証(手作り) | ユーザー登録・ログイン・ログアウト、TODO操作のログイン必須化 | 3h |
| Sprint 8 | 認可と負債返済 | user_id・Policyで「自分のTODOだけ」、FormRequest化(第1部で残した重複の回収)、教材最終化v2 | 3h |

## なぜこの順番か(第2部)

1. **軽い機能追加(5)が先**: 追加開発の型(影響調査→nullable追加→回帰確認)を小さく一周する
2. **テスト(6)を認証(7)より先に**: 認証は全ページに影響する大改修。それを「テストの傘の下」で
   行うことで、実務でテストを先に書く理由を体感する
3. **認証(7)と認可(8)を分ける**: 「ログインできる」と「他人のTODOは触れない」は別概念(401/403)。
   混同しやすいので意図的に分割
4. **リファクタリング(8)が最後**: 第1部で意図的に残したバリデーション重複をFormRequestで回収し、
   「テストがあればDRYの天秤は後から倒し直せる」で締める
