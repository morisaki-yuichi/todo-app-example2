# ユーザーストーリー

「〜として、〜したい、なぜなら〜」形式で、受け入れ条件(Acceptance Criteria)つきで記述します。
優先順位と実施スプリントは [product-backlog.md](product-backlog.md) で管理します。

> **ユーザーストーリーとは**: 機能を「作る側の作業」ではなく「使う人の欲求」として書く手法。
> 「なぜなら〜」を書くことで、実装中に迷ったとき立ち返る判断基準になります。
> なお US-0 のように開発者・写経者を主語にしたストーリーは、正確には
> 「テクニカルストーリー」と呼ばれることもあります。

---

## US-0: 開発環境の再現

**写経する初級エンジニアとして、READMEの手順どおりに環境を起動したい。
なぜなら、環境構築でつまずくと本編(CRUD実装)にたどり着けないから。**

### 受け入れ条件
- [ ] リポジトリをクローンした直後の状態から、READMEの手順だけで環境が起動できる
- [ ] `vendor/` がない状態でも起動できる手順(鶏と卵問題の解決)がREADMEにある
- [ ] `.env.example` からコピーした `.env` で `http://localhost:8080` にLaravelの画面が表示される
- [ ] ポートやプロジェクト名が他プロジェクトと衝突した場合の変更方法が説明されている

## US-1: TODOの一覧を見る

**TODOアプリの利用者として、登録したTODOを一覧で見たい。
なぜなら、やるべきことの全体を一目で把握したいから。**

### 受け入れ条件
- [ ] `/todos` でTODOが作成日時の新しい順に一覧表示される
- [ ] 各TODOのタイトルと完了状態が見える
- [ ] TODOが0件のときは「TODOがありません」等の案内が表示される(空一覧で壊れない)

## US-2: TODOの詳細を見る

**TODOアプリの利用者として、TODOの詳細(内容・作成日時)を見たい。
なぜなら、タイトルだけでは思い出せない作業のメモを確認したいから。**

### 受け入れ条件
- [ ] 一覧のタイトルをクリックすると詳細画面 `/todos/{id}` が開く
- [ ] タイトル・内容・完了状態・作成日時が表示される
- [ ] 存在しないIDにアクセスすると404が返る(500エラーにならない)

## US-3: TODOを作成する

**TODOアプリの利用者として、新しいTODOを登録したい。
なぜなら、やるべきことを忘れないうちに記録したいから。**

### 受け入れ条件
- [ ] 一覧から作成フォームに移動でき、タイトル(必須・100文字まで)と内容(任意・1000文字まで)を入力して保存できる
- [ ] 保存後は一覧にリダイレクトされ、作成成功のメッセージが表示される(PRGパターン)
- [ ] バリデーション違反時はフォームに差し戻され、エラーメッセージと入力済みの値が表示される
- [ ] リロードしても二重登録されない

## US-4: TODOを編集する

**TODOアプリの利用者として、TODOのタイトルや内容を修正したい。
なぜなら、書き間違いや状況の変化に合わせて情報を最新に保ちたいから。**

### 受け入れ条件
- [ ] 詳細(または一覧)から編集フォームに移動でき、現在の値が入力済みの状態で表示される
- [ ] 保存すると更新され、成功メッセージつきでリダイレクトされる
- [ ] バリデーションルールは作成時と同一に動作する

## US-5: TODOを完了/未完了にする

**TODOアプリの利用者として、TODOをワンクリックで完了にしたい(戻したい)。
なぜなら、編集フォームを開かずに進捗を記録したいから。**

### 受け入れ条件
- [ ] 一覧上の操作で完了⇔未完了を切り替えられる
- [ ] 完了したTODOは見た目で区別できる(打ち消し線等)
- [ ] 切り替えはGETではなく、データを変更するHTTPメソッドで行われる

## US-6: TODOを削除する

**TODOアプリの利用者として、不要になったTODOを削除したい。ただし誤操作は防ぎたい。
なぜなら、リストを整理したいが、うっかり消して後悔したくないから。**

### 受け入れ条件
- [ ] 削除前に確認ページが表示され、対象のタイトルを確認できる
- [ ] 確認ページで「削除する」を押すと削除され、成功メッセージつきで一覧に戻る
- [ ] 「キャンセル」で削除せずに戻れる
- [ ] 削除はGETリクエストでは実行できない

## US-8: TODOに期限日を設定する(第2部)

**TODOアプリの利用者として、TODOに期限日を設定したい。
なぜなら、いつまでにやるべきかを忘れたくないから。**

### 受け入れ条件
- [ ] 作成・編集フォームで期限日を任意入力できる(過去日も許可)
- [ ] 詳細・一覧に期限日が表示され、期限切れ(過去日かつ未完了)は一覧で強調表示される
- [ ] **期限なしの既存TODOはそのまま動き続ける**(追加開発の後方互換)

## US-9: TODOを絞り込む(第2部)

**TODOアプリの利用者として、状態やキーワードでTODOを絞り込みたい。
なぜなら、件数が増えると目当てのTODOを探すのが大変だから。**

### 受け入れ条件
- [ ] 状態(すべて/未完了/完了)とキーワード(タイトル・内容の部分一致)で絞り込める
- [ ] 絞り込み条件はURL(クエリパラメータ)に現れ、URLを共有すれば同じ結果が見られる
- [ ] 絞り込み結果が0件でも壊れない(案内文を表示)

## US-10: 一覧をページ分割する(第2部)

**TODOアプリの利用者として、一覧を数件ずつのページで見たい。
なぜなら、TODOが増えても一覧が縦に伸び続けると見通しが悪いから。**

### 受け入れ条件
- [ ] 一覧が5件/ページに分割され、ページ移動リンクが表示される
- [ ] 絞り込み条件を保ったままページ移動できる
- [ ] 存在しないページ番号や不正な値(page=abc)でもエラー画面にならない

## US-11: アカウントを作ってログインする(第2部)

**アプリの利用者として、アカウントを登録してログイン・ログアウトしたい。
なぜなら、自分専用のTODO管理の入り口を持ちたいから。**

### 受け入れ条件
- [ ] 登録フォームでname・email・password(確認つき)を入力してアカウントを作れる
- [ ] emailは重複不可、passwordは8文字以上・確認一致でないと差し戻される
- [ ] 登録済みのemail/passwordでログインでき、ログアウトもできる
- [ ] パスワードは平文ではなくハッシュで保存される

## US-12: 未ログインを弾く(第2部)

**アプリの運営者として、ログインしていない人にはTODOを見せたくない。
なぜなら、TODO機能は利用者向けの機能だから。**

### 受け入れ条件
- [ ] 未ログインで /todos 等にアクセスすると /login へリダイレクトされる
- [ ] ログイン中はヘッダーにユーザー名とログアウトボタンが表示される
- [ ] ※このスプリントでは全ログインユーザーがすべてのTODOを見る(所有権はUS-13/スプリント8)

## US-7: 開発を追体験する(教材)

**写経する初級エンジニアとして、コミット単位で開発の過程を追体験したい。
なぜなら、完成品のコードを眺めるだけでは「どの順で・なぜそう書くのか」が学べないから。**

### 受け入れ条件
- [ ] 開発トレースガイド(dev-walkthrough.md)が全コミットをカバーしている
- [ ] 各ステップに「足場の作り方(生成/編集/新規作成)」「動作確認(ブラウザ/CLI)」「ここでコミット」がある
- [ ] 概念解説集(laravel-concepts.md)が登場順に整備され、トレースガイドと相互リンクしている
- [ ] READMEから両ドキュメントへの導線がある
