### 1. TODOの詳細仕様

**Q**: TODOの項目・バリデーション・ソート順はどうするか?
**A**: シンプル3項目構成とする。初級教材はCRUDの本質に集中できる最小構成が最適という判断。

| 項目 | 型 | 制約 |
|---|---|---|
| title(タイトル) | 文字列 | **必須**・最大100文字 |
| description(内容) | テキスト | 任意・最大1000文字 |
| completed(完了フラグ) | 真偽値 | デフォルト `false` |

- **一覧のソート順**: 作成日時の新しい順(`created_at` 降順)
- **削除の確認ステップ**: **確認ページ方式**を採用
  - 理由1: 技術スタック制約が「Plain HTML/CSS(JS不使用)」のため、JSの `confirm()` は使えない
  - 理由2: 確認ページ方式は「GETでデータを変えない原則」「メソッドスプーフィング」を
    教材として扱う良い実例になる

### 2. 認証機能の要否

**Q**: ログイン機能をスコープに含めるか?
**A**: **含めない**。
- 4スプリントで「CRUD+写経可能な教材ドキュメント」を完成させるにはスコープ外が妥当
- Laravel Breeze は Tailwind / Alpine.js に依存するため「Plain HTML/CSS」制約とも相性が悪い

### 3. 写経者の学習時間(ベロシティの基準)

**Q**: 写経者が1スプリントに割ける学習時間の目安は?
**A**: **2〜3時間/スプリント**。
- 環境構築系スプリントは軽め(1.5〜2h)、実装スプリントは2.5〜3h程度に分量を調整する
- 週1スプリントのペースなら約1ヶ月で完走できる教材になる

### 4. GitHubリポジトリと運用ルール

**Q**: リポジトリの公開設定とブランチ戦略は?
**A**:
- **パブリックリポジトリ**とする(写経者がクローン・閲覧できる必要があるため)
- **ブランチ戦略**: `main` + 機能ブランチ(`feature/xxx`)+ Pull Request でマージ
  - 理由: 写経者にPRベースの開発フローを見せられる。
    開発トレースガイドの「GitHubのコミット/PRリンク」要件とも整合する
  - 開発開始前の初期ドキュメント(このファイル等)のみ、main直コミットとする
- **Sail関連の.env方針**(初期プロンプトの指示に基づく決定):
  - `APP_PORT=8080` / `VITE_PORT=5174` / `COMPOSE_PROJECT_NAME=todo-app-example2` を明示する
  - 理由: 他プロジェクトとポート・コンテナ名が衝突するのを防ぐため。
    値は `.env.example` に記載し、写経者が自分の環境に合わせて変更できることを明記する

## 2026-07-04 第2部(追加開発編)の開始と仕様決定

新規開発編(スプリント1〜4)完了後、**追加開発の流れを追体験できる教材**として
続編(スプリント5〜8)を開始する。教材には新規開発編になかった3要素を各ステップに加える:

1. **影響調査**: 着手前に「どのファイルを読み、何に影響するか」を調べる手順
2. **既存データへの配慮**: 稼働中テーブルへの変更(nullable等)、後方互換
3. **リグレッション確認**: 既存機能が壊れていないことのチェック

### スプリント5の仕様Q&A

| 論点 | 決定 |
|---|---|
| 期限日(due_date) | **任意項目・過去日も許可**(既存TODOは期限なしのまま=nullable)。一覧で期限切れを強調表示 |
| 一覧の並び順 | **現状維持**(作成日時の新しい順)。既存挙動を変えない=リグレッションリスク最小 |
| 絞り込み | 状態(すべて/未完了/完了)+キーワード(**タイトルと内容**の部分一致)。**GETフォーム**(見るだけの操作なのでGETが正解・URL共有可) |
| 教材の置き場所 | dev-walkthrough.md / roadmap.md に**第2部として追記**(1ファイルで通読可能に) |

### スクラムマスター裁量の決定(理由つき)

- **ページネーションは5件/ページ**: シーダーを12件に増やし、写経者が必ず複数ページを見られるようにする
- **日付入力はHTML標準の `<input type="date">`**: JS不使用の制約内でカレンダーUIが使える
- **バリデーションは `nullable|date`**: 過去日許可の決定に基づき日付形式のみ検証

スプリント1開始時に `ss -tlnp` でポートの空きを確認したところ、
**5174 は開発マシン上の別プロジェクトのコンテナが使用中**だった。
そのため `VITE_PORT=5180`(空きを確認済み)に変更する。

> まさに「ポートは他プロジェクトと衝突しうるから .env で明示する」方針の必要性を示す実例。
> 写経者も自分の環境で `ss -tlnp | grep <ポート番号>` で空きを確認してから値を決めること。
