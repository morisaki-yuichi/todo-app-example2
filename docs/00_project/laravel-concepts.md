# スプリント1で登場した概念

## 1. コンテナ/イメージ

**① 一言定義**: イメージは「アプリと実行環境一式を固めた設計図(読み取り専用)」、
コンテナは「イメージから起動した実行中のプロセス(隔離された箱)」。

**② なぜ必要か**: 「私のPCでは動くのに、あなたのPCでは動かない」事故を防ぐため。
PHPのバージョン違い、拡張モジュールの有無、OSの違い — 環境差はバグの温床です。
イメージに環境ごと閉じ込めれば、誰が起動しても同じ環境になります。
また、ホストに直接PHPやMySQLを入れないので、プロジェクトを消せば環境も綺麗に消せます。

**③ このリポジトリでの実例**:
- `mysql:8.4` イメージ → `todo-app-example2-mysql-1` コンテナ(DBサーバ)
- Sailがビルドする `sail-8.5/app` イメージ → `todo-app-example2-laravel.test-1` コンテナ(PHP実行環境)
- 確認コマンド: `./vendor/bin/sail ps`(→ [ステップ1-2](dev-walkthrough.md#ステップ1-2-envexample-をプロジェクト用に整備する))

## 2. Docker Composeとcompose.yaml

**① 一言定義**: 複数のコンテナ(アプリ+DBなど)を1つのファイル(`compose.yaml`)で
定義して、まとめて起動・停止する仕組み。

**② なぜ必要か**: コンテナを1個ずつ `docker run` で起動すると、ポート・ネットワーク・
起動順序などのオプションを毎回手で打つことになり、人によって起動方法がズレる事故が起きます。
compose.yamlに書いておけば「起動方法そのもの」をGitで共有できます。
さらに、Composeは**サービス名で名前解決できるネットワーク**を自動で作ります。

**③ このリポジトリでの実例**:
- `compose.yaml` に `laravel.test`(アプリ)と `mysql`(DB)の2サービスを定義
- `.env` の `DB_HOST=mysql` は IPアドレスではなく**サービス名**。
  アプリコンテナは「mysql」という名前でDBコンテナに接続している
  (→ [ステップ1-2](dev-walkthrough.md#ステップ1-2-envexample-をプロジェクト用に整備する))

## 3. ポートマッピング

**① 一言定義**: ホスト(あなたのPC)のポートとコンテナ内のポートを繋ぐ設定。
`8080:80` なら「ホストの8080番への通信をコンテナの80番へ転送」。

**② なぜ必要か**: コンテナは隔離されているので、繋がないとブラウザから届きません。
また、ホスト側の1つのポートは**1つのプロセスしか使えない**ため、複数プロジェクトを
動かすと衝突事故(`port is already allocated`)が起きます。だからこのプロジェクトでは
ポートを `.env` で明示し、変更可能にしています。

**③ このリポジトリでの実例**:
- `.env` の `APP_PORT=8080` → ブラウザで `http://localhost:8080`
- 実録: 予定していた `VITE_PORT=5174` が別プロジェクトと衝突し、5180に変更
  ([qa-log.md](qa-log.md))。空き確認は `ss -tlnp | grep 5174`

## 4. Composerと鶏と卵問題

**① 一言定義**: ComposerはPHPのパッケージ管理ツール。`composer.json`(欲しいものリスト)と
`composer.lock`(実際に入れたバージョンの記録)をもとに `vendor/` へライブラリを入れる。

**② なぜ必要か**: ライブラリを手でコピーして使うと、バージョンも出どころもバラバラになり、
セキュリティ修正も追えません(既知の脆弱性が残ったままのライブラリを使い続ける事故)。
`composer.lock` をコミットすることで「全員が全く同じバージョン」になります。
`vendor/` そのものは巨大なのでコミットせず、各自がComposerで復元します。
ここで**鶏と卵問題**が生まれます: このプロジェクトの `sail` コマンドは `vendor/` の中に
あるのに、`vendor/` を作るComposerはホストに入っていない —
解決策は「**Composer入りの使い捨てコンテナ**で最初の1回だけ `composer install` する」です。

**③ このリポジトリでの実例**:
- README「セットアップ手順 2」の `docker run ... laravelsail/php84-composer:latest composer install`
- 導入時も同じ考え方でインストーラをコンテナ内実行([ステップ1-1](dev-walkthrough.md#ステップ1-1-laravel本体をsail構成で導入する))

## 5. Laravel Sail

**① 一言定義**: Laravel公式の、Docker Compose一式を隠蔽してくれる開発環境。
実体は `compose.yaml` と、それを操作する薄いシェルスクリプト(`vendor/bin/sail`)。

**② なぜ必要か**: PHP・MySQL・拡張モジュールを全員が手で揃えるのは環境差事故のもと。
Sailなら `sail up -d` だけで揃います。押さえるべき仕組みは2つ:
- **.env駆動**: `APP_PORT` や `COMPOSE_PROJECT_NAME` など、compose.yamlの中の
  `${変数}` は `.env` から注入される。「設定は.envに集約」がSailの流儀
- **WWWUSERによる所有者問題の回避**: コンテナ内のプロセスがrootでファイルを書くと、
  ホスト側に「rootしか消せないファイル」ができる事故が起きる。sailスクリプトは
  起動時にホストのユーザーID(`WWWUSER`/`WWWGROUP`)をコンテナに渡し、
  コンテナ内でも同じIDでファイルを扱うことでこれを回避している
  (導入時の `docker run` に `-u "$(id -u):$(id -g)"` を付けたのも同じ理由)

**③ このリポジトリでの実例**:
- `./vendor/bin/sail up -d` で起動、コマンドは全て `./vendor/bin/sail artisan ...` に統一
- `.env` の `APP_PORT=8080 / VITE_PORT=5180 / COMPOSE_PROJECT_NAME=todo-app-example2`

## 6. .envと.env.example

**① 一言定義**: `.env` は「この環境だけの設定値(実物・Git管理外)」、
`.env.example` は「設定項目の見本(コミットする)」。

**② なぜ必要か**: 設定をコードに直書きすると、環境ごとに書き換えが必要になる上、
**DBパスワードやAPP_KEYのような秘密情報がGitHubに公開される事故**につながります
(公開リポジトリに漏れたAWSキーが数分で悪用される、は実際に多発している事故です)。
そこで「実物は各自の手元だけ、見本(項目名とダミー値)は共有」と分けます。
見本の整備を怠ると「クローンした人だけ動かない」事故になるため、
このプロジェクトでは **.env.example を正**とし、.envはそこからコピーして作ります。

**③ このリポジトリでの実例**:
- [ステップ1-2](dev-walkthrough.md#ステップ1-2-envexample-をプロジェクト用に整備する)で
  `.env.example` をMySQL・ポート明示に整備([コミット 1c97590](https://github.com/morisaki-yuichi/todo-app-example2/commit/1c97590))
- `.gitignore` に `.env` が入っていることも確認してみてください

## 7. artisan

**① 一言定義**: Laravel付属のコマンドラインツール。雛形生成(`make:...`)、
DB操作(`migrate`)、対話シェル(`tinker`)などをサブコマンドで提供する。

**② なぜ必要か**: ファイルを手で作ると、置き場所・クラス名・お作法が人によってズレます。
Laravelは「**設定より規約**」(決まった場所に決まった名前で置けば設定不要)の
フレームワークなので、規約に合ったファイルをジェネレータで作るのが最短で安全です。

**③ このリポジトリでの実例**:
- `./vendor/bin/sail artisan key:generate` / `sail artisan migrate`(スプリント1)
- スプリント2からは `make:model` などの雛形生成が登場予定

## 8. APP_KEY(アプリケーションの暗号化キー)

**① 一言定義**: Laravelがセッション・Cookie等を暗号化するための秘密鍵。
`php artisan key:generate` で生成され `.env` に保存される。

**② なぜ必要か**: 暗号化キーがなければCookie等を安全に扱えないため、Laravelは
起動を拒否します(500エラー)。また、**キーが漏れると暗号化Cookieを偽造される**
恐れがあるため、`.env` に置いてGit管理外にします。「キーをコミットしてしまい、
公開リポジトリで丸見え」は初学者が実際にやりがちな事故です。

**③ このリポジトリでの実例**:
- [実験1-A](dev-walkthrough.md#実験1-a-わざと失敗app_keyを空にして500を観察する)で
  わざと空にして500を観察 → ログ1行目 `No application encryption key has been specified.`

## 9. マイグレーション(導入編)

**① 一言定義**: データベースのテーブル構造の変更を、SQLではなくPHPのコードで
記述してバージョン管理する仕組み。`sail artisan migrate` で適用する。

**② なぜ必要か**: テーブルを手作業のSQLで作ると「誰がいつ何を流したか」が残らず、
開発者ごとにDBの形が違う事故が起きます。マイグレーションはGitで共有されるので、
クローンした人も `migrate` 一発で同じ構造を再現できます。

**③ このリポジトリでの実例**:
- スプリント1ではLaravel標準の3つ(users/cache/jobs)を適用しただけ
- 自分でテーブルを設計する話(up/down、カラム定義)は**スプリント2で詳しく扱います**

## 10. HTTPステータスコード(このプロジェクトで見る範囲)

**① 一言定義**: サーバがレスポンスに付ける3桁の結果コード。
2xx=成功、3xx=リダイレクト、4xx=クライアント側の問題、5xx=サーバ側の問題。

**② なぜ必要か**: 「なんか動かない」を機械的に切り分けるため。コードを見れば
**どちら側の・どんな問題か**が一瞬で絞れます。切り分けの第一歩として最強の手がかりです。

**③ このリポジトリでの実例**(登場したものから追記していきます):

| コード | 意味 | このリポジトリで見た場面 |
|---|---|---|
| 200 | 成功 | Welcome画面の表示確認(`curl -w "%{http_code}"`) |
| 500 | サーバ内部エラー | 実験1-AのAPP_KEY未設定 |
| (000) | ※HTTPコードではなくcurlの「接続不能」表示 | `.env`変更直後、開発サーバ再起動中にアクセスした瞬間 |

> 302(リダイレクト)、404(Not Found)、405(メソッド不一致)、419(CSRF)は
> スプリント2〜3で実物が登場した際に追記します。
