# スプリント6 バックログ(自動テストとCI)

## スプリントゴール

**「CRUDと絞り込みの主要な振る舞いをFeatureテストで自動検証でき、
PRを出すとGitHub Actionsがそのテストを自動実行する状態にする」**

対応するユーザーストーリー: [US-7(教材)](../00_project/user-stories.md) / PBI #18・#19
写経目安時間: **2.5〜3時間**

## このスプリントの位置づけ(なぜ今テストか)

スプリント5で「既存CRUD 5画面が200か」を**毎コミット手作業で確認**した。
この繰り返しこそ自動化すべきもの。そして次のスプリント7(認証)は
全ページに影響する大改修になる — その前に**テストの傘**を張っておくことで、
「壊したらテストが赤くなる」安全網の中で改修できる。

## 前スプリントのTryをこのスプリントで使う

- **T-10**: 手作業のリグレッション確認(既存CRUD)を、そのままFeatureテストに写経する
- **T-11**: 新機能(テスト・CI)導入前に前提(テストDB・Actionsの書式)を1分調べる
- **T-12**: 「まず落ちるテスト(red)を見てから緑にする」を徹底し、テストが機能する証拠を残す

## スコープ外

- Unitテスト網羅(モデルのisOverdue等は1つ入れる程度に留める)
- ブラウザ自動テスト(Dusk。JS不使用のためFeatureテストで十分)
- カバレッジ計測(教材の焦点ではない)

## リスクと対策

- **テストがtesting DBを汚す/本番DBを消す誤解** → RefreshDatabaseは
  phpunit.xmlの `DB_DATABASE=testing` に対して動く。開発DB(todo_app)とは別。
  この分離を概念解説で明示する
- CIの初回は失敗しがち(コンテナ・PHP版・DB待ち) → Actionsのログの読み方を教材化
