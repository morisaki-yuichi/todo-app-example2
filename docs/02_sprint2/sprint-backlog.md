# スプリント2 バックログ

## スプリントゴール

**「シーダーで投入したTODOが、一覧画面と詳細画面で閲覧できる(存在しないIDは404になる)」**

対応するユーザーストーリー: [US-1(一覧)、US-2(詳細)](../00_project/user-stories.md)
対応するPBI: #4(テーブルとモデル)、#5(シーダー)、#6(一覧)、#7(詳細)

写経目安時間: **2.5〜3時間**

## なぜ「表示」から作るのか(このスプリントの設計判断)

フォーム(Create)より先に一覧(Read)を作るのは、**部品が少ないから**。
データはシーダー(プログラムでダミーデータを入れる仕組み)で用意すれば、
ルート→コントローラ→モデル→ビューという**MVCの basic な通り道を最小構成で1周**できる。

## タスク一覧

| # | タスク | 完了条件 |
|---|---|---|
| T1 | スプリントバックログ作成(このファイル) | mainにコミットされている |
| T2 | 作業ブランチ `feature/sprint2-read` 作成 | ブランチ上で作業できる |

| T8 | 詳細画面+一覧からのリンク追加 | `/todos/{id}` が表示され、存在しないIDは404。一覧から遷移できる |
| T9 | トレースガイド・概念解説集のスプリント2分を執筆 | 各ステップが必須要素を満たす |
| T10 | PR作成(`git status` 綺麗を確認)→ マージ → マージ後のmainで動作確認 | クローン相当の手順(migrate→seed→閲覧)がmainで動く |
| T11 | レビュー記録・レトロ記録(**Tryを使った結果を必ず記録**) | 3件のTryの使用結果が書かれている |

## スコープ外

- 作成・編集・削除フォーム(スプリント3〜4)
- CSS(スプリント4)
- `/`(トップ)の扱い変更(現状はLaravelのWelcomeのまま。スプリント4の仕上げで整理)

## リスクと対策

- シーダーの日時が同一秒になり、`created_at` 降順の並びが不安定になる可能性
  → 表示確認ではタイトルの内容で確認し、必要なら並び順の第2キーを検討する
